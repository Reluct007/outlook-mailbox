# PRODUCT.md

## 产品定义

这个项目不是通用邮箱客户端。

它是一个面向大规模 Outlook 邮箱资源的 **实时邮件信号雷达**。

当前 V1 只解决一个非常具体的问题：

> 不再登录多个 Outlook 邮箱逐个翻未读，只看一个命中流就能处理当天工作。

系统存在的目的，是替代逐个打开指纹浏览器检查邮箱的人工流程。

---

## 当前目标用户

最核心用户是：

> 小团队内部运营同学

他们不是来“管理邮箱世界”的。

他们来这里是为了：

- 快速发现高价值新邮件
- 只看值得处理的命中
- 明确知道哪些 mailbox 正在失效、延迟或需要恢复

---

## 当前 V1 目标

### 业务目标

- 接入 Outlook Inbox
- 实时或近实时接收新邮件
- 识别高价值信号
- 生成命中事件
- 让命中流成为默认工作入口
- 让 mailbox 异常状态对运营可见

### 技术目标

- 以 Cloudflare-native 运行模型承载主链路
- 除 Postgres 外，尽量不依赖长期自管服务
- 在 webhook / lifecycle / recovery / renew 乱序下保持正确性
- 用 PG + R2 split 处理正文与查询压力

---

## 当前最重要的高价值信号

- 验证码邮件
- 包含以下关键词的邮件：
  - cashback
  - reward
  - redeem

---

## V1 工作流

V1 的默认工作流不是“看邮箱”。

而是：

```text
新邮件进入 Outlook Inbox
-> 系统识别高价值信号
-> 命中流出现事件
-> 运营同学处理命中
-> 异常 mailbox 在 health 摘要中暴露
```

---

## V1 核心页面 / 能力

### 1. 命中流

这是 V1 的主入口。

展示最近命中的邮件，字段包括：

- 命中时间
- 收件邮箱
- 发件人
- 主题
- 命中类型
- 命中值
- 规则名
- 置信度
- 是否已处理

### 2. Message detail

用于查看单条命中的上下文。

包含：

- message 基础字段
- preview / excerpt
- 可用时读取完整正文
- associated hits

### 3. mailbox health 摘要

V1 不做完整邮箱池控制台。

只保留最小必要的健康视图，暴露这些状态：

- healthy
- delayed
- recovery_needed
- recovering
- reauth_required
- disabled

---

## V1 最小筛选能力

V1 先支持命中流上的最小筛选：

- 时间范围
- 发件人
- 收件人
- hit type
- confidence
- processed 状态

## 核心原则

### 1. 首页只显示命中的邮件

首页是命中流，不是收件箱。

### 2. Recall 优先

宁可多报，不可漏报。

### 3. 规则是一等公民

信号通过规则定义，而不是只靠临时搜索。

### 4. Inbox Only

第一版只跟踪 Outlook Inbox。

### 5. operator visibility 是主链路的一部分

如果 mailbox 正在失败、延迟、恢复中，系统必须明确告诉运营同学。

不能 silent failure。

---

## 成功标准

V1 完成的判断标准不是功能数，而是：

1. 大幅减少打开指纹浏览器的次数
2. 不再需要频繁登录多个 Outlook 邮箱逐个翻未读
3. 命中流成为默认工作入口
4. mailbox 异常状态对运营同学清晰可见
5. 每天看邮件时间从 3 小时下降到 30 分钟以内

---

## 当前不追求的产品形态

当前阶段不追求：

- “完整 Mail Radar 平台”
- “通用邮件后台”
- “邮件运维控制台”
- “Cloudflare 化的旧 Go 系统翻版”

当前只追求：

> Outlook Inbox 新邮件  
> -> 信号识别  
> -> 命中流  
> -> 可处理工作台
