# DESIGN.md

## 设计真源

本文件是当前项目的唯一设计真源。

它定义的是：

- 这个产品首页到底是什么
- 用户进入后第一眼应该看到什么
- 哪些状态必须明确区分
- 交互和视觉上哪些规则不能退化

如果后续实现、文案或页面结构与本文件冲突，以本文件为准。

---

## 1. Product Surface

这个产品不是邮箱客户端。

它是一个跨多个 Outlook mailbox 聚合的 **OTP 面板**。

首页主任务只有一个：

> 打开  
> 看到最新 code  
> 点一下复制  
> 离开

邮件是运输层，不是主界面对象。

首页不应该先把用户带进“邮件列表”或“命中流后台”的心智。

---

## 2. Primary User Journey

默认用户旅程固定为：

```text
Open page
-> See latest code immediately
-> Copy
-> Leave
```

这意味着：

- 首页不是探索型页面
- 首页不是分析后台
- 首页不能把历史、异常、次级信号抬到和最新 code 同一级

---

## 3. Information Hierarchy

首页信息层级固定如下：

1. 一级：最新验证码主卡
2. 二级：状态区
3. 三级：历史验证码与次级信号
4. 四级：深入上下文和诊断

### 一级：最新验证码主卡

主卡必须直接显示：

- 最新 code
- 复制按钮
- 来源邮箱
- 收到时间
- 信号类型
- 必要时补充“跨 X 个 mailbox 的最新验证码”

首页第一屏必须是 **单一主卡**。

不是双主位，不是多卡并列，不是 dashboard 式总览。

### 二级：状态区

状态区只承担解释职责，不抢主位。

必须覆盖：

- `waiting_for_code`
- `delivery_path_unhealthy`

### 三级：历史与次级信号

这一区域可以展示：

- 最近几条历史验证码
- reward / cashback / redeem 等非 OTP 当前信号

但 OTP 永远优先。

### 四级：深入上下文

仅在需要时下钻：

- message detail
- mailbox diagnostics

---

## 4. State Model

首页至少要清楚表达 4 种状态：

### `ready`

已有最新验证码，可直接复制。

### `waiting_for_code`

当前没有新验证码，但链路健康。

它应该让用户感觉：

- 系统还在正常工作
- 只是现在还没有新 code
- 再等一会是合理动作

### `delivery_path_unhealthy`

当前没有可用验证码，而且邮件链路本身存在异常。

它应该让用户明确知道：

- 这不是普通等待
- 问题出在订阅、抓取、恢复或认证链路
- 应该进入恢复或诊断动作

### `empty`

系统还没有任何可展示的验证码历史，也没有当前 signal。

它不是错误，但也不是 waiting 的同义词。

### 状态表达规则

`waiting_for_code` 和 `delivery_path_unhealthy` 必须使用完全不同的界面语气。

不能只是同一种提示框换一段文案。

---

## 5. Interaction Rules

复制是首页唯一主动作。

因此交互规则必须非常明确。

### 复制成功

- 点击后按钮立即进入成功态
- 主卡内显示简短成功反馈，例如“已复制”
- 反馈不能依赖用户去看页面其他角落

### 复制失败

- 不能静默失败
- 必须在主卡内给出明确失败反馈
- 必须保留重试动作

### 重复点击

- 点击瞬间必须进入进行中或成功态
- 避免用户因为无反馈而连续点击

---

## 6. Trust Signals

这个工具最怕的不是不好看，而是不可信。

所以主卡内必须直接展示这些信任元信息：

- 来源邮箱
- 收到时间
- 当前信号类型
- 必要时“当前跨 X 个 mailbox 的最新验证码”

这些信息不能全部下沉到二级区。

用户必须在复制前就能完成“我信不信这个 code”的判断。

---

## 7. Responsive Behavior

Responsive 不是简单把桌面布局压成单列。

桌面端与移动端承担不同任务：

- 桌面端：`scan-and-copy`
- 移动端：`glance-and-tap`

### 桌面端

- 主卡占据明确主位
- 状态区紧邻主卡，但不抢主位
- 历史与次级信号继续下沉

### 移动端

- 主卡必须绝对置顶
- 复制按钮必须更大、更易点击
- 历史和异常区继续下沉
- 次级信号默认更克制，避免首屏过吵

---

## 8. Visual Direction

视觉语言采用 **工具感**。

不是后台感，不是消费 app 感，也不是通用 dashboard 感。

### 应该接近

- 登机牌
- 密码管理器的自动填充卡片
- 支付确认页

### 不应该接近

- AI 后台卡片网格
- 三栏概览页
- “欢迎来到你的邮箱信号中心”式 marketing 文案

### 视觉原则

- 高密度，但不乱
- 强对比
- 大号 code
- 少装饰
- 少 marketing 语气

---

## 9. Non-Goals

当前阶段明确不做：

- 完整邮箱客户端
- 全量命中流作为首页主入口
- reward / cashback / redeem 与 OTP 并列主卡
- 首页大段邮件 preview
- 首页先展示 message detail
- 通用邮件后台

---

## 10. Locked Decisions

本轮设计评审已锁定以下结论：

1. 首页第一屏采用单一主卡
2. `waiting_for_code` 与 `delivery_path_unhealthy` 使用完全不同的界面语气
3. 默认旅程固定为 `打开 -> 看到 -> 复制 -> 离开`
4. 视觉语言采用工具感
5. 桌面端与移动端采用不同优先级设计
6. 主卡必须直接显示信任元信息
7. 复制动作必须有明确成功/失败反馈

---

## 11. Implementation Guardrails

实现层在设计上必须服从以下约束：

- OTP 首页应走专用 read API
- 首页应基于跨 mailbox 聚合的 read model
- OTP 必须优先于其他 signal
- 非 OTP signal 只能作为次级信息出现
- 首页不能要求用户先理解 mailbox lifecycle 才能完成主任务

---

## 12. Success Criteria

设计完成后的判断标准不是“页面更像后台”。

而是：

1. 用户进入首页后，不需要点开 message detail 就能拿到最新 code
2. 用户可以在 5 秒内完成“判断可信 -> 复制 -> 离开”
3. 没有验证码时，用户能明确区分“等待中”与“链路异常”
4. 首页的默认动作是复制，不是浏览邮件
5. 页面不会被误认成命中流后台或通用 dashboard
